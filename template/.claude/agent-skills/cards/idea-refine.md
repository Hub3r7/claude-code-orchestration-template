# idea-refine — operating card

> Distilled from [`../idea-refine/SKILL.md`](../idea-refine/SKILL.md) at upstream `8c65303`. The full skill is canonical — on any conflict it wins. Regenerate on every vendored refresh.

**Purpose:** Act as an honest ideation partner: expand a raw idea divergently, stress-test it convergently, and ship a one-pager with explicit assumptions and trade-offs.

**Binding rules:**
- Restate the idea as a 'How Might We' statement, then ask 3-5 sharpening questions; do not proceed until user and success criteria are clear.
- Generate 5-8 considered variations (inversion, simplification, 10x, audience shift...) — never 20+ shallow ones.
- Before converging, name each direction's hidden assumptions: what you bet is true, what could kill it, what you ignore and why.
- Stress-test directions on user value (painkiller vs vitamin), feasibility, and differentiation — push back on weak ideas, no yes-machining.
- End with a one-pager: problem, direction, assumptions with validation steps, MVP scope, and a 'Not Doing' list making trade-offs explicit.
- Inside a codebase, ground variations in existing architecture and constraints — scan real files, reference specific patterns.

**Do NOT apply when:**
- The idea is already sharp with a chosen direction — go straight to planning/implementation, not re-ideation.
- The task is routine execution (bug fix, defined feature); no vague concept to expand or stress-test.
- The user needs requirement extraction via focused questioning rather than option generation — that is interviewing, not ideation.

**Go deep — read the full SKILL.md — when:**
- A full ideation/refinement session is the task itself — the skill holds the three-phase dialogue, lens catalog, and one-pager template.
- You need the extended ideation frameworks, evaluation rubric, or session examples referenced from the skill's companion files.
